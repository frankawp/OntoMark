# OntoMark V2

Ontology-Driven AI Native Wiki Builder

---

# 1. 项目目标

## 1.1 背景

当前知识库系统主要有两种模式：

### 文档模式

```text
目录
+
文件
```

例如：

```text
docs/

README.md

design.md

meeting.md
```

优点：

* 简单

缺点：

* 知识粒度不统一
* 概念重复出现
* 关系难以表达
* 不适合 AI

---

### RAG模式

```text
文档
→ Chunk
→ Embedding
→ Retrieval
```

优点：

* 部署简单

缺点：

* 每次重新检索
* 知识不会积累
* 跨文档关系弱
* 难以形成长期知识资产

---

## 1.2 OntoMark目标

OntoMark采用 Karpathy 提出的 LLM Wiki 思想。

核心理念：

```text
Compile Once
Query Many
```

不是：

```text
Query Time Synthesis
```

而是：

```text
Knowledge Compilation
```

即：

```text
Raw Sources
    ↓
Ontology
    ↓
Wiki
    ↓
Knowledge Graph
```

知识被编译成 Wiki。

后续查询直接使用 Wiki。

知识持续积累和演化。

# 3. 核心设计原则

## 3.1 Markdown First

Markdown 是唯一事实来源。

允许：

```text
Markdown

Frontmatter

WikiLink
```

禁止：

```text
entities.json

relations.json
```

作为长期存储。

缓存可重建。

---

## 3.2 Raw / Wiki 分层

遵循 LLM Wiki 三层架构：

```text
raw/
```

不可修改。

原始资料。

---

```text
wiki/
```

可修改。

知识编译层。

---

```text
ontology.yaml
```

知识组织规则。

Schema层。

---

## 3.3 Canonical Object

每个知识对象必须唯一。

例如：

```text
JWT
```

对应：

```text
wiki/Concepts/JWT.md
```

系统中只允许存在一个 Canonical Page。

Alias 允许多个。

Canonical Object 只允许一个。

---

## 3.4 Human-in-the-loop

实体发现可以自动完成。

实体合并必须允许人工参与。

禁止自动覆盖用户修改。

# 6. Ontology

## 6.1 Ontology作用

Ontology定义：

* 实体类型
* 分类体系
* 命名规则
* 关系规则

不定义具体实体。

---

## 6.2 默认Ontology

```yaml
entity_types:

  Topic:

  Concept:

  System:

  Component:

  ADR:

  Requirement:

  Incident:

  Team:

  Person:

  Tool:
```

---

## 6.3 查找规则

```text
./ontology.yaml

↓

./.ontomark/ontology.yaml

↓

~/.ontomark/ontology.yaml
```

# 7. Wiki Builder

## 7.1 Knowledge Object Discovery

目标：

识别哪些内容应成为独立知识对象。

例如：

```text
支付系统采用JWT认证。

ADR-001决定采用JWT。
```

发现：

```text
JWT

Payment System

ADR-001
```

---

## 7.2 Canonical Page Generation

生成：

```text
wiki/

Concepts/JWT.md
```

---

Frontmatter：

```yaml
---
tags:
  - Concept

status: canonical
---
```

---

## 7.3 Evidence Collection

所有页面必须记录来源。

例如：

```yaml
---
sources:
  - raw/design/auth.md
  - raw/adr/ADR-001.md
---
```

---

目的：

* 可追溯
* 可审计
* 可验证

---

## 7.4 Knowledge Consolidation

多个来源中的知识自动归并。

例如：

```text
设计文档
ADR
会议纪要
```

最终统一写入：

```text
JWT.md
```

而不是多个地方重复描述。

这是 OntoMark 的核心能力之一。

---

# 8. Entity Resolution

## 8.1 Alias

例如：

```text
JWT

JSON Web Token

JsonWebToken
```

统一为：

```text
JWT
```

---

自动生成：

```yaml
aliases:
  - JSON Web Token
```

---

## 8.2 Disambiguation

例如：

```text
Auth/JWT

Security/JWT
```

进入审核流程。

禁止自动合并。

---

# 11. Topic Builder

除了实体页。

还需要主题页。

例如：

```text
Authentication
```

不是实体。

而是：

```text
Topic
```

---

生成：

```text
wiki/Topics/Authentication.md
```

---

内容：

```text
Overview

Concepts

Systems

ADRs

Related Topics
```

---

作用：

知识地图（Map of Content）。

参考 LLM Wiki 的 Lint 思想。

## Structural

* 孤立页面
* 无反链页面
* 无来源页面

## Semantic

* 同义实体
* 命名冲突
* 关系冲突

---

## Knowledge

* 缺失定义
* 缺失来源
* 缺失主题页

# 16. 技术选型

## Runtime

```text
Node.js 22+
TypeScript 5+
```

* Claude Code生态兼容
* Codex生态兼容
* Markdown生态成熟

---

## CLI

```text
commander
```

---

```text
unified
remark
mdast
```

必须基于 AST。

禁止 Regex 解析 Markdown。

---

## Schema Validation

```text
zod
```

---

## Storage

长期存储：

```text
Markdown
```

缓存：

```text
SQLite
```

用途：

* Entity Cache
* Alias Cache
* Build Cache

删除后可重建。
